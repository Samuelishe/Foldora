# Project State

Дата: 2026-07-01.

Состояние: legacy context menu генерируется в HKCU под Foldora-owned roots, но видимый верхний пункт берётся из `FoldoraSettings.CreateFolderMenu.Title`. Текущий MVP-вид Explorer menu: `Создать папку -> optional one-level groups -> entries`, без лишнего видимого слоя `Foldora`; technical registry key остаётся `Foldora`. `FolderMenuEntry.GroupName` реализует простую одноуровневую группировку: пустое значение оставляет entry в root menu, непустое создаёт submenu с visible title группы. Entry/group registry key names остаются техническими; `GroupName` и `DisplayName` не используются как registry path. Entry keys получают registry `Icon = <entry.IconPath>`, если imported `.ico` существует. `foldora register-menu`, `foldora register-menu --dry-run`, `foldora register-menu --host-path`, legacy alias `--cli-path`, `foldora unregister-menu` и `foldora menu reset --yes` реализованы. Explorer registry commands должны запускать no-console `Foldora.MenuHost.exe`; `Foldora.Cli.exe` остаётся console tool для разработки и ручных команд. Команды `foldora apply --folder "<folder>" --icon "<absolute-icon-path>"`, `foldora apply --folder "<folder>" --entry-id "<entry-id>"`, `foldora create --target "<directory>" --entry-id "<entry-id>"` и `foldora clear --folder "<folder>"` работают через `desktop.ini`; команды `foldora menu list/add/remove/reset` управляют пользовательским списком меню. `menu add` поддерживает `--group "<group-name>"`, а `menu list` показывает group/root state. WPF editor реализует staged editing title/entries, add/remove draft entries, staged выбор `.ico`, прямой preview из `.ico` около 50x50 и controls для Explorer integration. UI cleanup phase 1 заменил технический `DataGrid` на карточки пунктов меню, скрывает `EntryId` из основного flow, разделяет обычную интеграцию и dangerous reset, а technical registry details показывает только в раскрываемом блоке. WPF grouping container UX показывает entries внутри visual containers `Без группы` и `<GroupName>`: non-empty group имеет editable header, inline delete-group confirmation, entry count, вложенные entry cards и contextual add-entry button. Entry cards теперь имеют compact view state и inline edit state; saved entries стартуют compact, новые draft entries стартуют раскрытыми, validation errors раскрывают affected card и показываются inline. `Без группы` не удаляется. `+ Добавить группу` создаёт первый draft entry в новой группе, но не вводит persistent empty groups или tree storage. Settings gear в custom title bar заменён на self-authored XAML vector icon без внешних ассетов. Импорт выбранных иконок в AppData происходит только по `Сохранить`; preview-файлы не генерируются. Если `ExplorerIntegrationEnabled = true`, WPF `Сохранить` после успешного settings save rebuild-ит Foldora-owned registry menu; если integration disabled, `Сохранить` пишет только settings.

Documentation consolidation: продуктовая концепция вынесена в `docs/PRODUCT_VISION.md`, целевой WPF/staged-save flow - в `docs/UX_FLOW.md`. Главный MVP-объект зафиксирован как пользовательский `FolderMenuEntry`; packs остаются будущим import/export механизмом.

Content area больше не дублирует application title `Foldora`; вместо этого используется semantic page header `Меню папок`/`Folder menu`. Startup exceptions теперь записываются в `%AppData%\Foldora\Logs\startup-error.log` и показываются через простой startup error dialog, чтобы окно не пропадало молча.

Repair/normalize command в MVP не реализован.

Technical debt foundation: `docs/TECH_DEBT.md` фиксирует active debt и accepted limitations. `TD-0001` повышен до high-priority research: текущая legacy context menu integration создаёт папку в правильной Desktop directory, но не получает original right-click cursor/icon-view coordinates через `%V`. Diagnostic prototype `diagnostics desktop-icon-position` вручную подтверждён: existing desktop icons можно двигать через Shell positioning, и Explorer grid displacement принят как acceptable behavior. После этого `Foldora.MenuHost` получил best-effort desktop placement для `create`: cursor screen position захватывается максимально рано, папка создаётся прежним Core flow, а если target directory равна user Desktop directory, созданный desktop item пытается reposition near captured cursor/menu selection point. После ручной проверки production path placement всё ещё мог не сработать, поэтому MenuHost теперь пишет JSONL diagnostics в `%AppData%\Foldora\Logs\menuhost-placement.log` и делает bounded retry только для race, когда Explorer desktop view ещё не видит newly-created item (`Desktop item was not found`). Placement failure non-fatal: успешное создание папки остаётся успешным. Подробный research snapshot находится в `docs/research/DESKTOP_ICON_PLACEMENT.md`. `TD-0002` переведён в `Cannot reproduce / Monitor`: текущая ручная проверка больше не воспроизводит first-created default icon, новые папки сразу показывают custom icon. Сейчас create/apply path не делает Shell refresh notification; random sleep, COM shell extension и exact original right-click positioning не добавляются без отдельного исследования.

Publish/dev layout foundation: `scripts/publish-dev.ps1` создаёт framework-dependent Release layout в `artifacts/publish/Foldora` и кладёт рядом стабильные executable paths:

```text
Foldora.App.exe
Foldora.Cli.exe
Foldora.MenuHost.exe
```

Script не регистрирует Explorer menu и не запускает приложение. При запуске WPF из publish-папки command-host resolver ищет sibling `Foldora.MenuHost.exe`; если host отсутствует, Explorer integration возвращает user-facing failure, а не регистрирует неправильный путь. CLI `register-menu --host-path` остаётся явным способом указать published `Foldora.MenuHost.exe`. `artifacts/` не входит в git.

Per-user install layout foundation: `scripts/install-user.ps1` переиспользует fresh dev publish output и копирует binaries в `%LocalAppData%\Programs\Foldora`, где `Foldora.App.exe`, `Foldora.Cli.exe` и `Foldora.MenuHost.exe` снова лежат рядом. При запуске installed WPF app resolver использует sibling `Foldora.MenuHost.exe`, поэтому Explorer registry command должен указывать на `%LocalAppData%\Programs\Foldora\Foldora.MenuHost.exe`. `Foldora.MenuHost.exe` не является сервисом, tray/background helper или autostart process; Explorer запускает его только по клику пользователя в Foldora legacy context menu. `scripts/uninstall-user.ps1` сначала пытается выполнить installed `Foldora.Cli.exe unregister-menu`, затем fallback-ом удаляет только Foldora-owned HKCU roots и удаляет install root. `%AppData%\Foldora` сохраняется по умолчанию, потому что settings/imported icons/logs являются пользовательскими данными, а существующие styled folders могут ссылаться на imported icons в `%AppData%\Foldora\icons`.

Следующий практический этап: ручная install smoke-проверка Explorer integration из `%LocalAppData%\Programs\Foldora` и best-effort desktop placement. Registry menu в install/manual проверке должен указывать на installed `Foldora.MenuHost.exe`, а не на Debug build output и не на console `Foldora.Cli.exe`.

Открытые вопросы:

- Точный UX для выбора custom style.
- Возможен ли user-grade UX создания desktop folder под курсором без COM/modern shell integration.
- Нужен ли небольшой Shell refresh notification abstraction после create/apply для `TD-0002`.
- Фактическое поведение Explorer placeholders `%1` и `%V` на Windows 11 после повторной ручной проверки.
- Preview generation/cache policy, если прямой WPF preview из `.ico` окажется недостаточным.
- Full tree migration, drag-and-drop group ordering и group icons остаются future work; текущий MVP ограничен one-level `GroupName`.
- Manual release packaging beyond per-user install layout.
- User-facing diagnostics для failures внутри `Foldora.MenuHost.exe`, когда он запускается из Explorer.
